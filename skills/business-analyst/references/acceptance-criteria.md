# 驗收標準（Acceptance Criteria）範例

## 格式建議

### Given-When-Then 格式

```markdown
### 驗收標準

#### 情境 1：[正常流程]

- **Given** [前提條件]
- **When** [使用者操作]
- **Then** [預期結果]

#### 情境 2：[錯誤處理]

- **Given** [前提條件]
- **When** [使用者操作]
- **Then** [預期結果]

#### 情境 3：[邊界條件]

- **Given** [前提條件]
- **When** [使用者操作]
- **Then** [預期結果]
```

---

## 完整範例：會員登入

```markdown
### 驗收標準

#### 情境 1：正確帳密登入

- **Given** 會員已完成註冊，帳號為 `john@example.com`，密碼為 `Pass123`
- **When** 使用者在登入頁面輸入正確帳號和密碼並點擊「登入」
- **Then** 系統顯示「登入成功」，並在 2 秒內跳轉至會員首頁

#### 情境 2：錯誤密碼

- **Given** 會員已完成註冊，帳號為 `john@example.com`，密碼為 `Pass123`
- **When** 使用者輸入正確帳號但錯誤密碼 `WrongPass`
- **Then** 系統顯示「帳號或密碼錯誤，請重新輸入」，留在登入頁

#### 情境 3：帳號不存在

- **Given** 會員尚未註冊
- **When** 使用者輸入未註冊的帳號 `nobody@example.com`
- **Then** 系統顯示「帳號或密碼錯誤，請重新輸入」

#### 情境 4：空白欄位

- **Given** 會員在登入頁面
- **When** 使用者點擊「登入」但未填寫任何欄位
- **Then** 系統顯示「帳號和密碼為必填欄位」

#### 情境 5：密碼隱藏顯示

- **Given** 會員在登入頁面的密碼輸入框
- **When** 使用者輸入密碼時
- **Then** 密碼以隱藏字元顯示（***）

#### 情境 6：登入嘗試次數限制

- **Given** 會員連續輸入錯誤密碼達 5 次
- **When** 使用者再次嘗試登入
- **Then** 系統鎖定帳號 30 分鐘，並顯示「帳號已鎖定，請 30 分鐘後再試」
```

## 非功能性需求範例

```markdown
### 效能需求

- 登入頁面載入時間不超過 2 秒
- 登入處理時間不超過 1 秒

### 安全需求

- 密碼傳輸使用 HTTPS
- 密碼雜湊儲存（不使用明文）
- 登入失敗超過 5 次鎖定帳號

### 相容性需求

- 支援 Chrome、Firefox、Safari、Edge 最新版本
- 支援手機和平板尺寸（響應式設計）
```

## 驗收標準 vs 測試案例

| 驗收標準 | 測試案例 |
|----------|----------|
| 描述「什麼是完成」 | 描述「如何測試完成」 |
| 給開發團隊的指引 | 給 QA 的執行指令 |
| 較高層次 | 較具體、可執行 |
| 可以口頭確認 | 需要實際執行 |

## 常見錯誤

### ❌ 錯誤：太模糊

```markdown
- 系統要能正常運作
- 使用者體驗要良好
```

### ✅ 正確：具體可測試

```markdown
- 使用者在 3 秒內收到登入結果回應
- 錯誤訊息要在輸入框下方顯示
```

## 驗收標準檢查清單

- [ ] 每個情境都是「可執行」的嗎？
- [ ] 每個結果都是「可觀察」的嗎？
- [ ] 涵蓋正常流程、錯誤處理、邊界條件了嗎？
- [ ] 有包含效能或安全需求嗎？（如適用）
- [ ] 團隊成員都能理解這些標準嗎？
