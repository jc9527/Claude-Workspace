# 時程估算方法

---

## 估算方法

### 1. 類比估算（Analogous Estimation）

**適用時機**：有類似專案經驗

```
時程 = 類似專案時程 × (新專案規模 / 舊專案規模)
```

**優點**：快速
**缺點**：準確度低

---

### 2. 參數模型（Parametric Model）

**適用時機**：有歷史資料

```
人月 = (需求量 × 生產力) / 人力數量
```

**優點**：有數學依據
**缺點**：需要準確的生產力數據

---

### 3. 三點估算（Three-Point Estimation）

**適用時機**：不確定性高

```
估計值 = (O + 4M + P) / 6

O = 樂觀估計
M = 最可能估計
P = 悲觀估計
```

**範例**：
- 樂觀（O）：10 天
- 最可能（M）：15 天
- 悲觀（P）：25 天
- 預估 = (10 + 4×15 + 25) / 6 = **15 天**

---

### 4. 階段產出估算

**適用時機**：需要細項規劃

| 階段 | 佔比 | 說明 |
|------|------|------|
| 需求分析 | 10-15% | 了解需求、訪談 |
| 系統設計 | 15-20% | 架構、介面設計 |
| 開發實作 | 40-50% | 主要coding |
| 測試 | 15-20% | 測試、修正 |
| 部署上線 | 5-10% | 部署、訓練 |

## 並行開發估算

### 人力配置原則

| 總工期 | 建議人力 |
|--------|----------|
| 1 個月 | 1-2 人 |
| 2-3 個月 | 2-3 人 |
| 3-6 個月 | 3-5 人 |
| 6-12 個月 | 5-10 人 |

### Brook's Law

> "往已經延遲的軟體專案加派人手，只會讓它更加延遲。"

**建議**：前期人力評估要準確，避免中途加人

---

## 時程 Buffer

### 建議 Buffer 比例

| 階段 | Buffer | 說明 |
|------|--------|------|
| 需求分析 | 10-15% | 需求變更 |
| 系統設計 | 10-15% | 設計調整 |
| 開發實作 | 15-20% | 技術難題、bug |
| 測試 | 15-20% | 發現問題 |
| 部署上線 | 10-15% | 環境問題 |

### 計算方式

```
緩衝時間 = 估算時間 × Buffer 比例
總時程 = 估算時間 + 緩衝時間
```
