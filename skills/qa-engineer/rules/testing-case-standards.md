# 測試案例產生與執行規範 (v2)

## 目的

每個專案啟動時，依據 WBS 識別測試項目，並產出完整的測試案例清單供強哥確認。確保測試結果真實可靠，避免 false positive。

---

## 測試層級定義

| 層級 | 說明 | 測試對象 | 負責人 |
|------|------|----------|--------|
| **UT** (Unit Testing) | 單元測試 | 函式、方法、類別 | Developer |
| **SIT** (System Integration Testing) | 系統整合測試 | API、模組之間的介接 | QA / Developer |
| **E2E** (End-to-End Testing) | 端對端測試 | 完整使用者流程 | QA Engineer |

---

## E2E 測試防呆機制（新增）

### 截圖前必須檢查

```
1. 先確認 HTTP 200
   - curl -I http://localhost:5000/
   - 若非 200，等 3 秒後重試，最多 3 次

2. 截圖後驗證內容
   - 使用 Playwright 檢查頁面關鍵文字
   - 例如：列表頁應包含「聯絡人」或「新增」
   - 若內容不符預期，判定 Fail，重新截圖
```

### E2E 失敗判定

| 情況 | 判定 |
|------|------|
| HTTP 404/500 | Fail |
| 截圖顯示錯誤頁面 | Fail |
| 頁面內容與預期不符 | Fail |
| 截圖時間超過 30 秒 | Timeout → Fail |

## 測試案例產生流程

```
需求輸入 → 識別 WBS → 識別測試點 → 產生測試案例 → 強哥確認 → 執行
```

## 測試案例清單格式

### 總表（Test Case Summary）

| 專案 | 測試項目標號 | 功能 | UT | SIT | E2E | 優先權 |
|------|--------------|------|-----|-----|-----|--------|
| P001 | TC-001 | 取得所有聯絡人 | ✅ | ✅ | ✅ | 高 |

---

### 分項測試案例格式

| 欄位 | 說明 |
|------|------|
| **測試案例編號** | TC-[流水號]，如 TC-001 |
| **功能** | 測試的功能名稱 |
| **測試層級** | UT / SIT / E2E |
| **測試類型** | 功能測試、效能測試、安全測試等 |
| **前置條件** | 執行前的必要條件 |
| **測試資料** | 使用的測試資料 |
| **測試步驟** | 詳細的執行步驟 |
| **預期結果** | 每步的預期輸出 |
| **優先權** | High / Medium / Low |
| **狀態** | 待確認 / 已確認 / 已執行 |

## 各層級測試維度

### UT（單元測試）

| 維度 | 說明 | 適用場景 |
|------|------|----------|
| **Function Test** | 驗證函式邏輯正確 | 有明確商業邏輯的函式 |
| **Boundary Test** | 邊界值測試 | 有數值範圍的輸入 |
| **Error Handling Test** | 錯誤處理 | 例外情況處理 |

### SIT（系統整合測試）

| 維度 | 說明 | 適用場景 |
|------|------|----------|
| **API Test** | API 端點測試 | REST API |
| **Data Integration Test** | 資料流測試 | JSON 讀寫、資料庫 |
| **Workflow Test** | 工作流程測試 | 多步驟串接 |

### E2E（端對端測試）

| 維度 | 說明 | 適用場景 |
|------|------|----------|
| **User Flow Test** | 使用者流程 | 完整操作流程 |
| **UI/UX Test** | 介面測試 | 網頁/桌面 UI |
| **Cross Browser Test** | 跨瀏覽器 | 不同瀏覽器 |
| **Smoke Test** | 冒煙測試 | 基本功能驗證 |

## 測試報告格式

### 每個測試項目報告包含

1. **測試案例總表**（Pass/Fail 統計）
2. **個別測試結果**（Pass / Fail / Blocked）
3. **附屬資料**
   - Log 檔案
   - 截圖（需通過內容驗證）
   - API Response
4. **問題記錄**

---

## 附屬資料命名規則

```
{專案編號}-{TC編號}-{維度}-{時間戳}.{副檔名}

範例：
P001-SIT-001-API-20260405_1800.log
P001-E2E-001-UI-20260405_1800.png
```

---

## 測試案例確認流程

```
1. QA Engineer 依據 WBS 產生測試案例清單
2. 列出所有測試項目（UT/SIT/E2E）
3. 強哥確認測試範圍和優先序
4. QA Engineer 執行確認後的測試
   - E2E：截圖前檢查 HTTP，截圖後驗證內容
5. 產出測試報告（含截圖/Log）
```

---

## Timeout 設定

| 任務類型 | 建議逾時 |
|----------|----------|
| 一般訊息回覆 | 60s |
| 開發任務（複雜） | 360s（6分鐘） |
| 簡單 Script | 120s |

**派工時使用 `timeoutSeconds` 參數指定**

---

## 注意事項

1. **每個 WBS 項目都應有對應的測試案例**
2. **UT 由 Developer 自行執行**
3. **SIT 和 E2E 由 QA Engineer 執行**
4. **E2E 截圖必須驗證內容，避免 false positive**
5. **所有測試結果都需記錄並回報**
6. **Fail 的項目需記錄問題並追蹤**
