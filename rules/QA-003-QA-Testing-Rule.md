# QA 測試規範

**版本：** v1.0
**日期：** 2026-04-06
**適用專案：** P003-FileManager-WASM 及未來所有專案

---

## 1. 核心理念

> **截圖當下就記錄，一次到位。不要一直操作 Word/DOCX 檔。**
> 只有在所有測試確認完成後，才一次性轉檔為 DOCX 報告。

---

## 2. 目錄結構

```
{project_root}/QA/
├── test_cases/                    # 每個測試案例一個資料夾
│   ├── {功能編號}_{功能名稱}/    # 例：FM001_資料夾瀏覽
│   │   ├── description.md        # 測試描述檔（必填）
│   │   └── screenshots/          # 截圖存放處
│   │       ├── 001_homepage.png   # 截圖（編號+描述）
│   │       ├── 002_dialog.png
│   │       └── ...
│   ├── {功能編號}_{功能名稱}/
│   │   └── ...
│   └── ...
├── test_summary.md                # 測試總表（所有案例一目了然）
└── scripts/                       # 測試腳本（Python 等）
    └── playwright_screenshot.py
```

---

## 3. description.md 格式

```markdown
# {功能編號}：{功能名稱}

## 測試資訊

| 欄位 | 內容 |
|------|------|
| 測試日期 | {YYYY-MM-DD} |
| 測試人員 | {姓名} |
| 功能編號 | {FM-001} |
| 功能名稱 | {資料夾瀏覽} |
| 測試環境 | {主機名稱} |

## 功能說明

{簡短描述這個功能做什麼}

## 測試步驟

1. {步驟一}
2. {步驟二}
3. {步驟三}

## 預期結果

- {預期結果一}
- {預期結果二}

## 實際結果

{描述實際發生了什麼}

## 截圖清單

| 編號 | 截圖檔案 | 說明 | 狀態 |
|------|---------|------|------|
| 001 | screenshots/001_homepage.png | 首頁 | ✅ |
| 002 | screenshots/002_dialog.png | 對話框彈出 | ✅ |

## 狀態判定

- ✅ PASS — 截圖符合預期結果
- ❌ FAIL — 截圖與預期不符
- ⏸️ PENDING — 待執行或待補測

## 備註

{任何需要記錄的事項}
```

## 4. test_summary.md 總表格式

```markdown
# {專案名稱} QA 測試總表

**測試日期：** {YYYY-MM-DD}
**測試人員：** {姓名}
**總案例數：** {N}
**通過：** {N} | **失敗：** {N} | **待補：** {N}

## 測試案例總表

| 編號 | 功能名稱 | API 測試 | UI 截圖 | 狀態 | 備註 |
|------|---------|---------|---------|------|------|
| FM-001 | 資料夾瀏覽 | ✅ | ✅ | ✅ PASS | |
| FM-002 | 新增資料夾 | ✅ | ❌ | ❌ FAIL | 對話框未截到 |
| ... | | | | | |

## 失敗案例

### ❌ FM-002：{功能名稱}

**問題：** {描述問題}
**原因：** {分析原因}
**對應方法：** {解決方案}
**負責人：** {誰負責補測}
**預計完成：** {日期}
```

## 5. 截圖命名規範

```
{編號}_{功能簡寫}_{描述}.png

例：
001_FM001_homepage.png
002_FM001_filelist_with_items.png
003_FM002_dialog_open.png
004_FM002_input_name.png
005_FM002_success.png
```

編號原則：
- **001, 002, 003** — 按操作順序遞增
- **一個案例多張截圖** — 每個關鍵步驟一張

## 6. 流程紀律

```
┌─────────────────────────────────────┐
│  1. 執行測試                         │
│     ├── API curl 測試                │
│     └── Playwright 截圖              │
│            ↓                         │
│  2. 建立案例資料夾 + description.md   │
│     └── 同時複製截圖進 screenshots/  │
│            ↓                         │
│  3. 填寫 description.md             │
│     ├── 測試步驟                    │
│     ├── 預期結果                    │
│     ├── 實際結果                    │
│     └── 截圖狀態                    │
│            ↓                         │
│  4. 更新 test_summary.md 總表        │
│            ↓                         │
│  5. 發現失敗 → 記錄原因             │
│     └── 下一輪補測                   │
│            ↓                         │
│  6. 全部確認 PASS                   │
│     └── 一次性轉 DOCX 報告          │
└─────────────────────────────────────┘
```

---

## 7. 轉 DOCX 時機

**嚴禁：** 每測完一個案例就轉一次 Word
**正確：** 所有測試案例完成 + 總表確認後，一次性轉檔
