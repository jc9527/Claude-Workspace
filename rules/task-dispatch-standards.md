# 任務派工規範 (v2)

## 目的

避免 Agent 任務 Timeout，提升派工成功率。

---

## 核心原則

### 🥇 第1優先：拆分任務（最有效）

**80% 的 Timeout 是因為任務太複雜**

| 錯誤做法 | 正確做法 |
|----------|----------|
| 一個 Agent 做 SD + Dev + QA | 分成 3 個 Agent 串聯 |
| 一個 Agent 產出完整報告 | 先生成內容，再另一個 Agent 轉報告 |
| QA 做測試 + 產報告 + 發送 | 測試、報告、發送分開 |
| 「一體機」任務 | 單一職責 Agent |

### 拆分原則

```
複雜任務 → 拆成多個簡單任務 → 串聯執行

範例：
❌ 錯誤：派一個 Agent 做「SD + Dev + QA + 報告」
✅ 正確：
  1. SD Agent → 設計文件
  2. Dev Agent → 實作（等 SD 完成）
  3. QA Agent → 測試（等 Dev 完成）
  4. QA Agent → 報告（等測試完成）
```

### 任務複雜度判斷

| 複雜度 | 特徵 | 派工方式 |
|--------|------|----------|
| **低** | 單一檔案、簡單操作 | 單一 Agent，120s |
| **中** | 多檔案、需要測試 | 單一 Agent，360s |
| **高** | 多步驟、需產報告、需跨模組 | 拆分任務，360s 每個 |
| **極高** | SD + Dev + QA + 報告 | 絕對拆分！ |

---

## Timeout 預設值

| 任務類型 | 預設 Timeout | 說明 |
|----------|--------------|------|
| 簡單訊息回覆 | 60s | 純文字回覆 |
| 檔案操作（讀寫） | 120s | 單一檔案 |
| 開發任務（單一） | 360s（6分鐘） | API、简单功能 |
| 複雜任務（多步驟） | 480s（8分鐘） | 需測試+報告 |
| 極複雜任務 | 拆分後執行 | 不要一次派 |

---

## 派工檢查清單

每次派工前檢查：

```
□ 任務是否太複雜？（SD+Dev+QA 是複雜）
□ 是否需要拆分？（複雜任務要拆分）
□ Timeout 是否夠長？（開發類 360s 起）
□ 環境是否準備好？（套件、目錄）
□ 是否有明確的產出定義？（產出什麼檔案）
```

## 任務拆分範例

### 範例 1：聯絡人系統開發

| 步驟 | Agent | Timeout | 依賴 |
|------|--------|---------|------|
| 1. SD 設計 | System Designer | 360s | 無 |
| 2. 後端實作 | Developer | 360s | SD 完成 |
| 3. 前端實作 | Developer | 360s | SD 完成 |
| 4. SIT 測試 | QA Engineer | 360s | Dev 完成 |
| 5. E2E 測試 | QA Engineer | 360s | Dev 完成 |
| 6. 報告產出 | QA Engineer | 180s | 測試完成 |

### 範例 2：產出測試報告

| 步驟 | Agent | Timeout | 依賴 |
|------|--------|---------|------|
| 1. 執行測試 | QA Engineer | 300s | 無 |
| 2. 收集資料 | QA Engineer | 60s | 測試完成 |
| 3. 產出報告 | QA Engineer | 120s | 資料收集完成 |
| 4. 發送報告 | 主控 | 60s | 報告完成 |

---

## 常見 Timeout 情境與解決

| 情境 | 原因 | 解決 |
|------|------|------|
| SD+Dev+QA 同時做 | 太複雜 | 拆分串聯 |
| 產出完整 Word 報告 | 內容太多 | 先生成內容再轉報告 |
| 安裝多個套件 | 時間長 | 預先安裝好 |
| 大量檔案讀寫 | IO 時間長 | 減少檔案數或延長 Timeout |

---

## 總結

1. **複雜任務一定要拆分**
2. **Timeout 預設 360s 起**
3. **單一職責 Agent 優於一體機**
4. **派工前檢查清單**
